## 1. 변동성 필터 무효화

### 위치
`src/strategy/indicators.py` — `calc_volatility_ratio()` (line 150–153)

### 현상
변동성 비율 계산 시 데이터가 부족하면 **기본값 `1.0`을 반환**한다.  
`1.0`은 "변동성 정상" 판정 기준이므로, 데이터 부족 = 필터 항상 통과.

### 로그 증거
```
전체 4,266줄 중 약 3,000줄이 "변동성 계산 데이터 부족" 경고
02:10 ~ 02:39 사이에만 700회 이상 반복
```

### 근본 원인
- `long_window = 192`개 캔들이 필요
- `candle_count = 200`개만 수집 (config.py)
- 신규 상장 코인은 192개 미만의 15분봉 데이터만 존재
- 결과적으로 **대부분의 코인에서 변동성 필터가 작동하지 않음**

### 영향
변동성이 극단적으로 높은 코인(AZTEC, SXP 등)에 무방비 진입 허용 → 손절 발생

### 개선 방안
| 방안 | 설명 |
|------|------|
| **A. 안전한 기본값** | 데이터 부족 시 `1.0` 대신 `999.0` (매우 높은 변동성) 반환 → 진입 차단 |
| **B. 캔들 수 증가** | `candle_count`를 `500`으로 상향하여 충분한 데이터 확보 |
| **C. 최소 데이터 요건** | 필요한 캔들 수의 80% 미만이면 해당 코인 진입 금지 |

**권장**: A + C 조합. 데이터 부족 자체를 위험 신호로 취급해야 함.

## 2. 잦은 재시작으로 BB 이탈 추적 상태 유실

### 위치
`src/strategy/engine.py` — `_was_below_lower` (line 49)

### 현상
볼린저 밴드 하단 이탈 여부를 추적하는 `_was_below_lower` 딕셔너리가 **메모리에만 존재**한다.  
봇이 재시작되면 이 상태가 초기화되어, BB 이탈 없이도 매수 신호가 발생할 수 있다.

### 로그 증거
하루(02:10 ~ 23:30) 동안 **최소 17회 재시작** 확인:
```
02:10, 02:15, 02:48, 09:51, 15:18, 16:13, 16:33, 16:34, 
16:42, 16:50, 17:50, 18:04, 21:36, 21:59, 22:05, 22:11, 23:24
```

### 영향
- 재시작 직후 `_was_below_lower` 전부 `False`로 초기화
- BB 하단 이탈 이력이 없는 코인에 대해 매수 신호가 잘못 발생 가능
- 특히 16:33~16:50 사이 5분 간격 반복 재시작 구간에서 상태 불안정

### 개선 방안
| 방안 | 설명 |
|------|------|
| **A. DB 영속화** | `_was_below_lower` 상태를 Supabase `bot_state` 테이블에 저장 |
| **B. 시작 시 복구** | 봇 시작 시 최근 캔들로 BB 이탈 상태를 역산하여 복구 |
| **C. 프로세스 안정화** | 잦은 재시작의 근본 원인(에러 핸들링, 메모리 등) 해결 |

**권장**: B + C. 시작 시 캔들 데이터로 상태를 복원하고, 재시작 빈도 자체를 줄여야 함.

## 3. Supabase RLS 권한 오류

### 위치
전체 백엔드 — Supabase 클라이언트 호출부 전반

### 현상
다수의 Supabase 테이블에서 RLS(Row Level Security) 권한 오류가 반복 발생한다.

### 에러 로그 상세
| 테이블 | 에러 유형 | 발생 횟수 | 시간대 |
|--------|----------|----------|--------|
| `system_logs` | permission denied | 3회 | 산발적 |
| `bot_state` | table not found → permission denied | 20+회 | 11:48~11:59 연속 |
| `trades` | permission denied | 2회 | 16:33, 16:34 |
| `kakao_tokens` | permission denied | 다수 | 알림 시도 시 |
| `balance_snapshots` | permission denied | 5회 | 21:39~22:08 |

### 핵심 영향
```
[CRITICAL] 봇 초기화 실패 2회 (16:33, 16:34)
  → trades 테이블 접근 불가로 봇 시작 자체 실패
  → 재시작 루프 발생
```

- **DB 기록 실패**: 거래 이력, 시스템 로그 누락
- **봇 시작 실패**: trades 테이블 조회 불가 시 CRITICAL 에러
- **알림 불가**: kakao_tokens 접근 불가로 카톡 전송 실패
- **자산 추적 불가**: balance_snapshots 기록 불가

### 개선 방안
1. Supabase 대시보드에서 각 테이블의 RLS 정책 확인 및 수정
2. 서비스 키(service_role key) 사용 여부 확인 — 봇은 서버사이드이므로 service_role 키 사용이 적절
3. 테이블 존재 여부 확인 (`bot_state` 테이블이 실제 생성되었는지)

## 4. 저유동성 코인 체결 타임아웃

### 위치
`src/executor/order_executor.py` — `_wait_for_fill()` (line 169–202)

### 현상
시장가 주문이 체결되지 않고 타임아웃되는 현상이 반복된다.

### 로그 증거
| 시간 | 종목 | 상황 |
|------|------|------|
| 10:00 | ORCA | 매수 타임아웃 |
| 12:34 | XRP | 매수 타임아웃 |
| 17:00 | SXP | 매수 타임아웃 1회 |
| 17:25 | AZTEC | 매수 타임아웃 1차 — 주문 즉시 취소됨 |
| 17:26 | AZTEC | 매수 타임아웃 2차 — 주문 즉시 취소됨 |
| 17:28 | AZTEC | 매수 타임아웃 3차 — 주문 즉시 취소됨 |

### 근본 원인
- 시장가 주문임에도 유동성 부족으로 거래소가 자동 취소
- `_wait_for_fill`이 300초(5분) 대기 — 시장가 주문은 즉시 체결/취소되므로 비효율적
- 주문 접수 직후 "이미 취소된 주문" 응답 → 체결 불가 상태에서 5분 대기

### 영향
- 5분 대기 동안 다른 기회 놓침
- AZTEC 3회 연속 실패에도 계속 재시도 (쿨다운 없음)

### 개선 방안
| 방안 | 설명 |
|------|------|
| **A. 타임아웃 단축** | 시장가 주문 대기 시간을 300초 → 30초로 축소 |
| **B. 즉시 취소 감지** | "이미 취소된 주문" 응답 시 즉시 실패 처리 |
| **C. 실패 쿨다운** | 동일 코인 체결 실패 시 일정 시간(30분) 재시도 금지 |
| **D. 유동성 필터** | 호가 스프레드 또는 체결량 기준 최소 유동성 확인 후 주문 |

**권장**: A + B + C. 빠른 실패 + 재시도 방지가 핵심.

## 5. SXP 더스트 반복 실패 (수정 완료)

### 위치
`src/orchestrator.py` — `_execute_sell_half()`

### 현상 (수정 전)
SXP 보유량이 소액(~6,302원)인 상태에서 절반 매도 시도 시, 절반 금액(~3,151원)이 업비트 최소 주문 금액(5,000원) 미만이어서 반복 실패.

### 로그 증거
```
23:27 ~ 23:30 사이 8회 연속 동일 에러:
"최소 주문 금액(5000 KRW) 미달: 3156 KRW"
"최소 주문 금액(5000 KRW) 미달: 3156 KRW"
... (6회 더 반복)
```

### 수정 내용
`_execute_sell_half()`에 3단계 분기 로직 적용:
1. **절반 ≥ 5,000원** → 정상 절반 매도
2. **전량 ≥ 5,000원 & 절반 < 5,000원** → 전량 매도로 전환
3. **전량 < 5,000원** → 더스트로 판정, 포지션 정리 (매도 생략)

### 현재 상태
✅ **수정 완료** — `orchestrator.py`에 반영됨. 런타임 검증 필요.

## 7. 전략 로직 결함

### 7-1. RSI 과매도 기준 느슨

**위치**: `src/strategy/engine.py`

**현상**: RSI 과매도 판정 기준이 `rsi_oversold + 10 = 40` 이하이면 매수 허용.

```python
# 현재 로직 (의사코드)
if rsi <= rsi_oversold + 10:  # 30 + 10 = 40
    allow_buy = True
```

**문제**: AZTEC이 RSI=35.7에서 매수 허용 → 하락 추세 중 진입 → -7,449원 손절.  
RSI 40은 과매도라기보다 **약세 구간**에 해당하며, 반등 확률이 충분히 높지 않음.

**개선**: `rsi_oversold + 5 = 35` 또는 `rsi_oversold` 자체(30)로 엄격화.

### 7-2. 추세 필터 부재

**현상**: 현재 전략은 추세 방향을 고려하지 않음. 하락 추세에서도 평균 회귀(볼린저 밴드 하단 터치) 매수 진행.

**문제**:  
- 강한 하락 추세에서 BB 하단은 지지선이 아닌 **하락 가속 신호**
- AZTEC: BB 하단 터치 → 매수 → 추가 하락 → 손절

**개선**:
- 이동평균선(MA20/MA50) 배열 확인: MA20 > MA50일 때만 매수
- 또는 ADX(Average Directional Index) 추가: 추세 강도 필터

### 7-3. 수수료 미반영 익절 기준

**현상**: 익절 기준이 수수료를 고려하지 않아, 0.1% 이하 수익의 "익절"이 실제로는 손실.

**증거**: XRP 1차 익절 (2,094 → 2,093 = -0.05%), SXP 익절 (46 → 44.9 = -2.4%)

**개선**: 최소 익절 마진을 수수료(0.1%) + 알파(0.2%) = 최소 0.3% 이상으로 설정.

### 7-4. get_krw_balance() None 처리 누락

**위치**: `src/orchestrator.py`

**현상**: `get_krw_balance()`가 `None`을 반환할 때 TypeError 크래시 발생.

**로그 증거**:
```
[14:53] TypeError: '>' not supported between instances of 'NoneType' and 'int'
```

**개선**: None 반환 시 기본값 0 처리 또는 재시도 로직 추가.

### 7-5. 전종목 OHLCV 개별 조회 비효율

**위치**: `src/collector/data_collector.py` — `get_top_volume_symbols()`

**현상**: 모든 KRW 마켓 코인의 OHLCV를 개별 API 호출로 조회 → rate limit 위험 + 약 3분 소요.

**개선**: 업비트 Ticker API(`/v1/ticker`)로 전종목 거래대금을 한 번에 조회 후 상위 N개만 OHLCV 수집.
